# Technical Specification

---

## О Проекте
Мы разрабатываем CMS для сети спешелти-кофеен. Бариста используют это приложение на десктопах или планшетах, чтобы просматривать профили зерна и стандарты заваривания.

**Цели:**
1.  Построение чистой архитектуры: `Controller -> Service -> Repository`.
2.  Имитация базы данных через файловую систему (JSON).
3.  Использование индустриального формата локализации **XLIFF 2.0**.
4.  Строгая типизация в TypeScript.

---

## Технический Стек
*   **Runtime:** Node.js.
*   **Framework:** Express.js.
*   **Language:** TypeScript.
*   **Storage:** JSON-файлы в папке `/data` (NoSQL подход).
*   **Frontend:** Vanilla JS.
---

## Этапы Разработки

Проект разделен на две стадии.

### ЭТАП 1: Alpha (MVP)
*Обязательная часть.*

1.  Доступ к файлам изолирован в слое репозитория (Repository Pattern).
2.  Создание, чтение, обновление и удаление карточек кофе.
3.  Перевод интерфейса на IT и BG языки (XLIFF 2.0).
4.  Подключить библиотеку `morgan` в режиме `dev` (HTTP Logging).
5.  Картинки выбираются из списка флагов. Рецепты создаются автоматически.

### ЭТАП 2: Beta 
*Дополнительная часть для самостоятельного развития проекта.*

*   Замена JSON-файлов на реальную SQL базу данных.
*   Локализация *контента* через БД (хранение переводов описаний в базе).
*   Загрузка реальных фото через `multer` вместо выбора флагов.
*   Детальное редактирование шагов рецепта.
*   **Menu & Categories:**
    *   Добавить поле `type`: `"bean"`, `"beverage"`, `"dessert"`.
    *   Реализовать фильтрацию: `GET /api/beans?type=beverage`.
    *   Добавить табы-переключатели на фронтенде.

## Бизнес-логика

В этом разделе описаны правила, по которым работает Бэкенд.

### 4.1. Управление Карточками

**1. Создание (Create / POST)**
*   Проверить наличие обязательных полей `title`, `country`. Если нет — `400 Bad Request`.
*   Сервер сам генерирует уникальный ID (UUID v4).

**House Standard**

Так как в базовой форме бариста вводит только описание зерна, система **обязана** автоматически добавить техническую карту заваривания. Если в теле запроса нет массива `recipes`, сервер добавляет массив с двумя дефолтными рецептами (см. код ниже).

**2. Обновление (Update / PUT)**
*   Форма редактирования на фронтенде (Alpha) не отправляет массив рецептов. При обновлении (`PUT`) бэкенд должен:
    1.  Прочитать текущий файл с диска.
    2.  Извлечь из него существующий массив `recipes`.
    3.  Объединить его с новыми данными, пришедшими с фронтенда.
    4.  Перезаписать файл.

**3. Удаление (Delete)**
*   Файл физически удаляется с диска `fs.unlink`.

### 4.2. Локализация 

*   Весь контент в файлах `.json` хранится на **Английском языке**. Он не переводится.
*   **Интерфейс:** Переводятся только элементы UI.
*   **Обработка запроса:**
    *   Клиент запрашивает: `GET /api/i18n/it`
    *   Сервер ищет файл: `locales/it.xlf`
    *   Если файл найден: Парсит XML -> Конвертирует в JSON -> Отдает клиенту.
    *   Если файл не найден: Вернуть ошибку 404 или фоллбэк на английский.

### 4.3. Работа с Изображениями (Alpha)
*   Фронтенд посылает строку пути (ссылку), например: `assets/flags/ke.svg`.
*   Бэкенд сохраняет эту строку в поле `imageUrl` "как есть", не проверяя физическое наличие файла.

## Спецификация API

### Карточки Кофе (Beans)

*   `GET /api/beans` — Список всех лотов.
*   `GET /api/beans/:id` — Детальная карточка.
*   `POST /api/beans` — Создание (см. Бизнес-логику п.4.1).
*   `PUT /api/beans/:id` — Обновление (см. Бизнес-логику п.4.1).
*   `DELETE /api/beans/:id` — Удаление.

### Локализация

*   `GET /api/i18n/:lang`
    *   Сервер читает файл `locales/{lang}.xlf` (XLIFF 2.0).
    *   Возвращает JSON-объект вида `{"key": "translation"}`.

**Пример XLIFF 2.0 (`locales/it.xlf`):**
```xml
<xliff xmlns="urn:oasis:names:tc:xliff:document:2.0" version="2.0" srcLang="en" trgLang="it">
  <file id="f1">
    <unit id="ui.myBeans">
      <segment>
        <source>My Beans</source>
        <target>I Miei Caffè</target>
      </segment>
    </unit>
  </file>
</xliff>
```

## Стартовый набор

Вы получаете архив `project-starter.zip`, содержащий:
1.  Папку `public` — готовый Frontend (HTML/CSS/JS).
2.  Папку `data` — примеры JSON и XLIFF файлов.
3.  Файл `js/api.js` внутри фронтенда содержит пустые функции. Вам нужно реализовать бэкенд так, чтобы эти функции могли получать данные.

**Что нужно знать:**
1.  Фронтенд ничего не знает о файлах. Он просто шлет запросы на `http://localhost:3000/api`.
2.  **`js/api.js`:** в этом файле находятся пустые функции-заглушки. Задача написать там `fetch` запросы к своему серверу.
3.  **Modal Form:** в форме создания есть выпадающий список "Country Flag". Он отправляет на сервер строку  `assets/flags/ke.svg`. Сервер просто сохраняет эту строку в поле `imageUrl`.

**Ваши первые шаги:**
1.  Инициализировать проект `npm init`, `tsc --init`.
2.  Установить зависимости `express`, `cors`, `uuid`, `fast-xml-parser` и их `@types`.
3.  Создать структуру папок `src/controllers`, `src/services`, `data/beans`.
4.  Открыть `public/js/api.js` и посмотреть, какие методы ожидает фронтенд.
